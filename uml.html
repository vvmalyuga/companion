UML моделирование системы «Компаньон»

Unified Modeling Language (UML) — стандартизированный язык графического описания для объектного моделирования в области разработки программного обеспечения. Ниже представлены 9 ключевых диаграмм UML, описывающих различные аспекты системы.

Типы UML диаграмм в проекте:

  1. Диаграмма прецедентов (Use Case) — взаимодействие пользователей с системой
  2. Диаграмма классов (Class Diagram) — структура данных системы
  3. Диаграмма последовательностей (Sequence Diagram) — временные взаимодействия
  4. Диаграмма объектов (Object Diagram) — экземпляры классов в runtime
  5. Диаграмма компонентов (Component Diagram) — физическая архитектура
  6. Диаграмма развертывания (Deployment Diagram) — распределение по серверам
  7. Диаграмма ролей/акторов (Roles/Actors) — роли пользователей
  8. Диаграмма очередей/взаимодействий (Queue/Interaction) — потоки сообщений
  9. Диаграмма системной динамики (System Dynamics) — изменение состояния во времени

1. Use Case Diagram (Диаграмма прецедентов)

Поведенческая диаграмма

Описание:

Диаграмма показывает взаимодействие различных акторов (пользователей) с системой «Компаньон». Отображает основные сценарии использования системы.

Основные акторы:

  • Заказчик: Регистрируется, ищет компаньона, бронирует встречи, оплачивает, оставляет отзывы
  • Компаньон: Создает профиль, управляет расписанием, принимает заявки, получает оплату
  • Администратор: Модерирует пользователей, управляет контентом, просматривает статистику
  • Платежная система: Внешняя система обработки платежей

Ключевые прецеденты:

Прецедент Актор Описание
UC-01: Регистрация Заказчик, Компаньон Создание учетной записи с выбором роли
UC-02: Поиск компаньона Заказчик Поиск по фильтрам и категориям
UC-03: Бронирование встречи Заказчик Выбор даты, времени и оформление заявки
UC-04: Принятие заявки Компаньон Подтверждение или отклонение бронирования
UC-05: Оплата услуг Заказчик Оплата через интегрированную платежную систему
UC-06: Оставление отзыва Заказчик Написание отзыва и оценка после встречи
UC-07: Модерация контента Администратор Проверка и одобрение/отклонение контента

2. Class Diagram (Диаграмма классов)

Структурная диаграмма

Описание:

Диаграмма показывает статическую структуру системы: классы, их атрибуты, методы и взаимосвязи между ними.

Основные классы системы:

Класс: User (Пользователь)

Атрибуты: id, email, password_hash, role, created_at

Методы: register(), login(), updateProfile()

Наследование: Родительский класс для Customer и Companion

Класс: Customer (Заказчик)

Атрибуты: preferences, payment_methods, subscription_type

Методы: searchCompanion(), bookMeeting(), leaveReview()

Ассоциации: Имеет множество Booking

Класс: Companion (Компаньон)

Атрибуты: bio, hourly_rate, categories, availability

Методы: updateSchedule(), acceptBooking(), withdrawFunds()

Ассоциации: Имеет множество Booking и Review

Класс: Booking (Бронирование)

Атрибуты: id, date, time, duration, status, total_price

Методы: confirm(), cancel(), reschedule()

Ассоциации: Принадлежит Customer и Companion

Класс: Review (Отзыв)

Атрибуты: id, rating, comment, date, is_verified

Методы: publish(), edit(), report()

Ассоциации: Принадлежит Customer, относится к Companion

Класс: Payment (Платеж)

Атрибуты: id, amount, currency, status, transaction_id

Методы: process(), refund(), getReceipt()

Ассоциации: Связан с Booking один-к-одному

Типы отношений между классами:

  • Наследование (Inheritance): Customer и Companion наследуют от User
  • Ассоциация (Association): Customer имеет множество Booking
  • Агрегация (Aggregation): System содержит множество Users
  • Композиция (Composition): Booking состоит из Payment
  • Зависимость (Dependency): Review зависит от Booking

3. Sequence Diagram (Диаграмма последовательностей)

Поведенческая диаграмма

Описание:

Диаграмма показывает временную последовательность взаимодействия объектов в рамках конкретного сценария использования.

Сценарий: Бронирование встречи

  1. Customer (Заказчик): Выбирает компаньона и нажимает "Забронировать"
  2. Frontend: Отправляет запрос на бронирование в API
  3. BookingController: Проверяет доступность выбранного времени
  4. Database: Возвращает информацию о доступности
  5. BookingController: Создает предварительную запись о бронировании
  6. PaymentService: Инициирует процесс оплаты
  7. Payment Gateway: Обрабатывает платеж и возвращает результат
  8. NotificationService: Отправляет уведомление компаньону
  9. Companion (Компаньон): Получает уведомление о новой заявке
  10. Frontend: Отображает подтверждение бронирования заказчику

Альтернативные потоки:

  • Время недоступно: Система предлагает альтернативные слоты
  • Оплата не прошла: Бронирование отменяется автоматически
  • Компаньон отклоняет заявку: Средства возвращаются заказчику

4. Object Diagram (Диаграмма объектов)

Структурная диаграмма

Описание:

Диаграмма показывает конкретные экземпляры классов (объекты) и связи между ними в определенный момент времени.

Конкретные объекты в системе:

Объект: customer1

Класс: Customer

Атрибуты: id = "cust_001", name = "Анна Иванова", email = "anna@mail.com"

Связи: Имеет booking1, booking2

Объект: companion1

Класс: Companion

Атрибуты: id = "comp_001", name = "Максим Петров", rate = 1500₽/час

Связи: Принимает booking1, имеет review1

Объект: booking1

Класс: Booking

Атрибуты: id = "book_001", date = "2025-03-15", time = "18:00"

Связи: Принадлежит customer1, назначен companion1

Объект: review1

Класс: Review

Атрибуты: id = "rev_001", rating = 5, comment = "Отличная компания!"

Связи: Написан customer1, относится к companion1

Значение для тестирования:

Диаграмма объектов помогает тестировщикам создавать конкретные тестовые сценарии, основанные на реальных данных и связях между объектами.

5. Component Diagram (Диаграмма компонентов)

Структурная диаграмма

Описание:

Диаграмма показывает физические компоненты системы и зависимости между ними.

Основные компоненты системы:

Web Client

Тип: Веб-браузер

Зависимости: Загружает ресурсы с Web Server

Содержит: HTML, CSS, JavaScript файлы

Web Server

Тип: Nginx / Apache

Функции: Обслуживание статических файлов, балансировка нагрузки

Порты: 80 (HTTP), 443 (HTTPS)

Application Server

Тип: Node.js / Python

Модули: User Management, Booking Engine, Payment Processor

API: REST endpoints для клиентов

Database Server

Тип: PostgreSQL, MongoDB

Базы: users_db, bookings_db, payments_db

Репликация: Master-Slave для отказоустойчивости

External Services

Платежный шлюз: ЮKassa, Stripe

СМС сервис: Twilio, SMS.ru

Email сервис: SendGrid, Mailgun

Облачное хранилище: AWS S3, Yandex Object Storage

Интерфейсы между компонентами:

  • WebClient ↔ WebServer: HTTP/HTTPS протокол
  • WebServer ↔ AppServer: Reverse proxy (порт 3000)
  • AppServer ↔ Database: TCP соединение, SQL/NoSQL запросы
  • AppServer ↔ ExternalServices: REST API, Webhooks

6. Deployment Diagram (Диаграмма развертывания)

Структурная диаграмма

Описание:

Диаграмма показывает физическое размещение компонентов системы на аппаратных узлах.

Аппаратная архитектура:

Узел: Load Balancer

Спецификации: 2 CPU, 4 GB RAM

Компоненты: HAProxy, Nginx

Назначение: Распределение трафика между серверами

Узел: Web Servers (x2)

Спецификации: 4 CPU, 8 GB RAM каждый

Компоненты: Nginx, статические файлы

Назначение: Обслуживание веб-контента

Узел: App Servers (x3)

Спецификации: 8 CPU, 16 GB RAM каждый

Компоненты: Node.js, бизнес-логика

Назначение: Обработка запросов, выполнение операций

Узел: Database Cluster

Спецификации: 16 CPU, 64 GB RAM

Компоненты: PostgreSQL (master), MongoDB (sharded)

Назначение: Хранение и обработка данных

Узел: Cache Server

Спецификации: 4 CPU, 8 GB RAM

Компоненты: Redis, Memcached

Назначение: Кэширование часто запрашиваемых данных

Узел: File Storage

Спецификации: Облачное решение

Компоненты: AWS S3 совместимое хранилище

Назначение: Хранение фотографий и файлов пользователей

Сетевые соединения:

  • Публичная сеть: Load Balancer ↔ Интернет (80/443 порты)
  • Приватная сеть: Все серверы в одном VLAN
  • База данных: Репликация между узлами по защищенному каналу
  • Резервное копирование: Ежедневные backup в удаленное хранилище

7. Roles / Actors Diagram (Диаграмма ролей/акторов)

Поведенческая диаграмма

Описание:

Диаграмма детально описывает роли пользователей в системе, их права, обязанности и взаимодействие.

Иерархия ролей и прав доступа:

Роль Права доступа Ограничения Типичные действия
Гость (неавторизованный) Просмотр главной страницы, чтение информации о сервисе Нет доступа к поиску, бронированию, профилям Регистрация, просмотр отзывов
Заказчик (обычный) Поиск компаньонов, просмотр профилей, бронирование, оплата Не может создавать анкету компаньона Бронирование встреч, написание отзывов
Заказчик (премиум) Все права обычного заказчика + приоритет в поиске, расширенные фильтры Дополнительная оплата подписки Расширенный поиск, приоритетные уведомления
Компаньон (базовый) Создание анкеты, управление расписанием, прием заявок Ограниченное количество фото в профиле Обновление профиля, подтверждение бронирований
Компаньон (верифицированный) Все права базового + отметка "Проверено", повышенный рейтинг Требуется прохождение верификации Верификация личности, повышение доверия
Компаньон (премиум) Все права верифицированного + выделение в поиске, аналитика Ежемесячная плата за премиум-статус Продвижение профиля, доступ к статистике
Модератор Проверка профилей, модерация отзывов, блокировка пользователей Не может изменять системные настройки Проверка контента, ответы на жалобы
Администратор Полный доступ ко всем функциям, управление системой Ответственность за безопасность системы Настройка системы, управление пользователями

Переходы между ролями:

  • Гость → Заказчик/Компаньон: После регистрации и подтверждения email
  • Заказчик обычный → премиум: После оплаты подписки
  • Компаньон базовый → верифицированный: После успешной верификации личности
  • Компаньон верифицированный → премиум: После оплаты премиум-статуса
  • Пользователь → Модератор: По назначению администратором

8. Queue / Interaction Diagram (Диаграмма очередей/взаимодействий)

Поведенческая диаграмма

Описание:

Диаграмма показывает потоки сообщений и взаимодействий между компонентами системы, включая очереди и асинхронную обработку.

Очереди сообщений в системе:

Очередь: Booking Requests

Назначение: Обработка запросов на бронирование

Производитель: Frontend при отправке формы бронирования

Потребитель: Booking Service

Максимальный размер: 10 000 сообщений

Очередь: Email Notifications

Назначение: Отправка email уведомлений

Производитель: Notification Service при событиях

Потребитель: Email Service

Задержка: До 5 минут для неважных уведомлений

Очередь: Payment Processing

Назначение: Обработка платежных транзакций

Производитель: Payment Service при создании платежа

Потребитель: Payment Gateway Integration

Гарантия доставки: At-least-once delivery

Очередь: Review Moderation

Назначение: Модерация отзывов пользователей

Производитель: Review Service при публикации отзыва

Потребитель: Moderation Service

Время обработки: До 24 часов

Паттерны взаимодействия:

  • Request-Reply: Синхронные запросы к API (например, проверка доступности)
  • Publish-Subscribe: Уведомления о событиях (например, новая заявка)
  • Message Queue: Асинхронная обработка длительных операций
  • Event Sourcing: Сохранение всех изменений состояния для аудита

Технологии реализации:

  • Message Broker: RabbitMQ или Apache Kafka
  • Протоколы: AMQP для очередей, WebSocket для реального времени
  • Сервисы: Микросервисная архитектура с четкими границами

9. System Dynamics Diagram (Диаграмма системной динамики)

Диаграмма моделирования

Описание:

Диаграмма показывает динамику изменения состояния системы во времени, причинно-следственные связи и обратные связи.

Ключевые переменные системы:

R(t) — Зарегистрированные пользователи

Уравнение: dR/dt = α + δ * Z(t)

Где: α = органический рост (100 чел/мес), δ = реферальный коэффициент (0.05)

A(t) — Активные пользователи

Уравнение: dA/dt = β * R(t) + ρ * A(t) - γ * A(t)

Где: β = активация (0.3), ρ = удержание (0.2), γ = отток (0.1)

M(t) — Назначенные встречи

Уравнение: dM/dt = ε * A(t) - η * M(t)

Где: ε = конверсия в бронирования (0.4), η = завершение встреч (0.9)

Z(t) — Завершенные встречи

Уравнение: dZ/dt = η * M(t)

Назначение: Показатель успешных взаимодействий

Обратные связи в системе:

  • Положительная обратная связь: Успешные встречи (Z) приводят к новым регистрациям (R) через рекомендации
  • Отрицательная обратная связь: Перегрузка системы приводит к снижению активности пользователей
  • Балансирующая петля: Система рейтингов регулирует качество компаньонов
  • Усиливающая петля: Сетевой эффект — больше пользователей привлекает еще больше пользователей

Прогноз развития системы:

Период Пользователи Встречи/мес Выручка, ₽/мес Примечания
Месяц 1 (запуск) 500 100 120 000 Начальная аудитория, тестирование
Месяц 6 5 000 1 500 1 800 000 Стабильный рост, сетевой эффект
Месяц 12 15 000 5 000 6 000 000 Лидер рынка, расширение в регионы
Месяц 24 50 000 20 000 24 000 000 Международная экспансия, новые услуги
