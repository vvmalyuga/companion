Фреймовое моделирование системы

Фрейм-модели (frame models) — это метод структурного представления знаний о предметной области, где информация организуется в виде фреймов (кадров). Каждый фрейм описывает определенный концепт системы через слоты (атрибуты) и их значения.

Преимущества фреймового подхода:

  • Структурированность: Четкая организация знаний
  • Иерархичность: Возможность наследования свойств
  • Модульность: Независимое описание компонентов
  • Гибкость: Легкость расширения и модификации
  • Наглядность: Визуальное представление структуры

Примеры фрейм-моделей системы

1. Фрейм "Контент" (Frame Content)

Структура фрейма:

Фрейм: Content
Слоты:
  - id: Уникальный идентификатор
  - type: Тип контента (profile, review, photo)
  - owner: Владелец контента (User.id)
  - created_at: Дата создания
  - updated_at: Дата обновления
  - status: Статус (draft, published, archived)
  - visibility: Видимость (public, private, friends)
  - metadata: Дополнительные метаданные
  
Наследование:
  Content → ProfileContent
  Content → ReviewContent
  Content → PhotoContent
  
Пример объекта:
  Content_001:
    id: "cont_001"
    type: "profile"
    owner: "user_123"
    created_at: "2025-01-15 10:30:00"
    status: "published"
    visibility: "public"

2. Фрейм "Роль" (Frame Role)

Структура фрейма:

Фрейм: Role
Слоты:
  - role_id: Идентификатор роли
  - role_name: Название роли
  - permissions: Список разрешений
  - description: Описание роли
  - hierarchy_level: Уровень в иерархии
  - parent_role: Родительская роль
  - default_settings: Настройки по умолчанию
  
Конкретные роли:
  Role_Customer:
    role_id: "role_001"
    role_name: "Заказчик"
    permissions: ["search", "book", "review"]
    hierarchy_level: 1
    
  Role_Companion:
    role_id: "role_002"
    role_name: "Компаньон"
    permissions: ["create_profile", "accept_booking", "receive_payment"]
    hierarchy_level: 1
    
  Role_Admin:
    role_id: "role_003"
    role_name: "Администратор"
    permissions: ["all"]
    hierarchy_level: 3
    parent_role: null

3. Фрейм "Класс" (Frame Class)

Структура фрейма:

Фрейм: Class
Слоты:
  - class_name: Имя класса
  - attributes: Атрибуты класса
  - methods: Методы класса
  - parent_class: Родительский класс
  - interfaces: Реализуемые интерфейсы
  - access_modifier: Модификатор доступа
  - is_abstract: Абстрактный класс?
  
Пример класса User:
  class_name: "User"
  attributes:
    - id: String
    - email: String
    - password_hash: String
    - created_at: DateTime
  methods:
    - register(): Boolean
    - login(): Session
    - logout(): Void
    - updateProfile(): Boolean
  parent_class: null
  is_abstract: false

Наследование:
  User → Customer
  User → Companion

Дополнительные фрейм-модели

Фрейм "Бизнес-процесс"

Фрейм: BusinessProcess
Слоты:
  - process_id: Идентификатор процесса
  - process_name: Название процесса
  - description: Описание процесса
  - steps: Шаги выполнения
  - actors: Участники процесса
  - inputs: Входные данные
  - outputs: Выходные данные
  - exceptions: Исключительные ситуации
  
Пример: Process_Booking
  process_id: "proc_001"
  process_name: "Бронирование встречи"
  steps: ["select_companion", "choose_time", "confirm_booking", "process_payment"]
  actors: ["Customer", "Companion", "PaymentSystem"]
  inputs: ["companion_id", "datetime", "customer_id"]
  outputs: ["booking_id", "payment_confirmation"]

Фрейм "Данные"

Фрейм: Data
Слоты:
  - data_id: Идентификатор данных
  - data_type: Тип данных
  - format: Формат данных
  - size: Размер данных
  - source: Источник данных
  - destination: Назначение данных
  - sensitivity: Чувствительность данных
  - retention_period: Период хранения
  
Пример: CustomerProfileData
  data_id: "data_001"
  data_type: "structured"
  format: "JSON"
  size: "2-5 KB"
  source: "User registration form"
  destination: "Users database"
  sensitivity: "PII (Personal Identifiable Information)"
  retention_period: "5 years after account deletion"

Фрейм "Взаимодействие"

Фрейм: Interaction
Слоты:
  - interaction_id: Идентификатор взаимодействия
  - from: Отправитель
  - to: Получатель
  - message_type: Тип сообщения
  - content: Содержание
  - timestamp: Временная метка
  - status: Статус доставки
  - response_required: Требуется ответ?
  
Пример: ChatMessage
  interaction_id: "msg_001"
  from: "user_123"
  to: "user_456"
  message_type: "text"
  content: "Привет! Когда можем встретиться?"
  timestamp: "2025-01-15 14:30:00"
  status: "delivered"
  response_required: true

Иерархия фреймов системы «Компаньон»

Основные категории фреймов:

Уровень 1: Сущности (Entities)

  • User (Пользователь)
    • Customer (Заказчик)
    • Companion (Компаньон)
    • Admin (Администратор)
  • Content (Контент)
    • Profile (Профиль)
    • Review (Отзыв)
    • Photo (Фотография)

Уровень 2: Процессы (Processes)

  • RegistrationProcess (Процесс регистрации)
  • SearchProcess (Процесс поиска)
  • BookingProcess (Процесс бронирования)
  • PaymentProcess (Процесс оплаты)

Уровень 3: Данные (Data)

  • UserData (Данные пользователя)
  • BookingData (Данные бронирования)
  • PaymentData (Платежные данные)
  • AnalyticsData (Аналитические данные)

Уровень 4: Взаимодействия (Interactions)

  • UserSystemInteraction (Взаимодействие пользователь-система)
  • UserUserInteraction (Взаимодействие пользователь-пользователь)
  • SystemExternalInteraction (Взаимодействие система-внешние сервисы)

Применение фрейм-моделей в разработке

Проектирование базы данных

Фреймы "Класс" и "Объект" напрямую преобразуются в таблицы БД и записи в них.

Пример: Фрейм User → Таблица users в PostgreSQL

Разработка API

Фреймы "Взаимодействие" определяют форматы запросов и ответов REST API.

Пример: Фрейм BookingProcess → POST /api/bookings endpoint

Создание интерфейса

Фреймы "Роль" и "Контент" определяют, какие элементы интерфейса видны разным пользователям.

Пример: Фрейм Role_Companion → Кнопка "Принять заявку" в интерфейсе

Тестирование

Фреймы "Прецедент" служат основой для тестовых сценариев и use case тестирования.

Пример: Фрейм RegistrationProcess → Тест регистрации нового пользователя
